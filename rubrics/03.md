## タスク: Excel数式のPandas変換
## task_type: fact
## weights: logic_and_fact=60, constraint_adherence=30, helpfulness_and_creativity=10

## 評価目的

Excel の入れ子条件を、評価順序と境界値を変えずに Pandas の `DataFrame` へ移し、
新しい `Status` 列を生成する正しいコードを提示できるかを評価する。特定 API への一致ではなく、
同じ truth table を実現するかを重視する。

## 採点者向け事実情報

Excel 式の意味は次の通りである。先に成立した分岐が採用される。

1. `A >= 80` かつ `B == "Member"` なら `VIP`。
2. それ以外で、`A >= 50` または `B == "Member"` なら `Regular`。
3. それ以外は `Guest`。

境界例:

- `A = 80`, `B = "Member"` は `VIP`。
- `A < 80`, `B = "Member"` は `Regular`。
- `A >= 50`, `B != "Member"` は `Regular`。
- `A < 50`, `B != "Member"` は `Guest`。

`np.select`、入れ子の `np.where`、段階的な `.loc`、正しい `apply` などは、結果が同じなら妥当な別実装である。

## Critical Fail Conditions

| ID | 条件 |
| --- | --- |
| CF-1 | 最終的に推奨するコードが `AND` と `OR`、分岐順序、または `VIP` / `Regular` / `Guest` の対応を取り違え、上記 truth table の中心部分を誤る。 |
| CF-2 | 最終的に推奨する唯一の実装が、構文エラー、存在しない主要 API、未定義の中心変数などにより実行不能で、回答内に動作する代替実装がない。 |

不足した import や説明用の省略が、周辺文脈から容易に補えるだけの場合は CF-2 にせず通常減点とする。

## 評価ルーブリック

### 1. Logic & Fact（0〜60点）

#### 評価観点

- `VIP` を最優先にし、その残りへ `Regular` 条件を適用しているか。
- `>= 80`、`>= 50`、`Member` の境界とラベルが正しいか。
- `&` / `|` を使う場合、各比較を括弧で囲むなど Python / Pandas の演算規則を正しく扱うか。
- 回答中のサンプル出力がある場合、コードの truth table と一致するか。

#### 得点アンカー

| 得点帯 | 基準 |
| --- | --- |
| 54〜60 | 全分岐と境界が正しく、優先順位も明示またはコード構造から明確。出力例があれば整合する。 |
| 39〜53 | 中心ロジックは正しいが、括弧、欠損値の扱い、説明または出力例に局所的な不備がある。 |
| 18〜38 | 一部のケースは正しいが、分岐順序や境界の理解が不安定で、特定ケースで誤る。 |
| 0〜17 | truth table の中心が誤る、または正しい実装を特定できない。 |

### 2. Constraint Adherence（0〜30点）

#### 評価観点

- Pandas の `DataFrame` を対象にし、新しい列 `Status` へ結果を代入しているか。
- A 列を購入額、B 列を会員ランクとして扱い、求められた変換以外へ論点を逸らしていないか。
- 提示コードが一つの首尾一貫した実装として使用可能か。

#### 得点アンカー

| 得点帯 | 基準 |
| --- | --- |
| 27〜30 | `df["Status"]` 等へ正しく代入する、実行可能で首尾一貫した Pandas コードを提示する。 |
| 20〜26 | Pandas と `Status` 列の要件は満たすが、import、列名、変数名、コード断片の接続に小さな補完が必要。 |
| 9〜19 | ロジックの説明だけで列追加コードがない、Pandas 以外へ置き換える、または複数案が互いに矛盾する。 |
| 0〜8 | 指定環境・列追加要件をほぼ満たさない。 |

### 3. Helpfulness & Creativity（0〜10点）

#### 評価観点

- コードが過不足なく、ユーザーの DataFrame へ移植しやすいか。
- 分岐の順序、演算子の括弧、欠損値や型の前提など、実際に誤りやすい点を必要な範囲で説明しているか。
- データ量を考えた場合にベクトル化された実装を優先するなど、Pandas として妥当な選択か。

#### 得点アンカー

| 得点帯 | 基準 |
| --- | --- |
| 9〜10 | 簡潔でそのまま移植しやすく、分岐順序または代表的な落とし穴を短く説明する。ベクトル化または同等に妥当な実装。 |
| 7〜8 | 実用的なコードだが、説明、効率、前提のいずれかが弱い。 |
| 3〜6 | 正しさは読み取れるが、冗長な行ループ、過度な boilerplate、または補完作業が多い。 |
| 0〜2 | コードが断片的で使い方が分からない、または有用な実装がない。 |

## 軸間の切り分け

- truth table、境界、演算子の意味は `logic_and_fact` で評価する。
- Pandas、`DataFrame`、`Status` 列、コードの一貫性は `constraint_adherence` で評価する。
- 移植しやすさ、効率、落とし穴の説明は `helpfulness_and_creativity` で評価する。
- 構文エラーを Constraint と Logic の双方で同じ理由のまま重複減点しない。CF-2 でない局所エラーは、実行可能性へ最も直接対応する軸で主に扱う。
