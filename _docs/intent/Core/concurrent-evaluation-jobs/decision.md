# Intent: Concurrent evaluation jobs with provider rate limits

## Context

評価 run は設定（被験モデル・judge・タスク・パラメータ）ごとに結果を分けて持ちたい需要が
ある。現状は 1 本完走してから次を起動する運用になり、待ち時間が直列化する。同時に複数本を
走らせると、同一 API キー／プロバイダへリクエストが重なり、429 やコスト急増が起きやすい。
本 Intent は、**最大 3 ジョブの同時実行**、**進行ボード一式＝1 ジョブの縦積み UI**、
**プロセス共有のプロバイダ別レート制限（Settings 編集可・推奨デフォルト内蔵）** を固定する。

## Decisions

### DEC-001: 同時実行の単位は評価ジョブ（SSE `/run` 1 本）とし、上限は 3

- **What**: 設定一式の評価セッションを 1 ジョブとする。同一プロセスで同時に走れるジョブ数の
  上限は 3 とし、サーバが開始時に拒否する。フロントも同じ上限で開始を止める。
- **Why**: 2〜3 本並列が用途の主帯であり、無制限同時起動はレート制限以前に UI・コスト・
  観測負荷が崩れる。
- **Change freedom**: 上限の数値（3）は Settings 化してよいが、初版の既定とサーバ強制は
  維持する。拒否時のエラー形は変更できる。
- **Why not**: 「タスク並列だけ増やす」では設定違いの比較ジョブを並べられない。
- **Revisit when**: 夜間バッチ等で 3 を超える同時実行が常態化した時。

### DEC-002: Run 画面は進行ボード一式を 1 ジョブとして縦に積む

- **What**: 現行の進行ボード（待機 / 実行中 / 完了レーンとジョブ要約）を 1 ジョブのコンテナ
  とし、最大 3 つを同一画面に縦積みする。キャンセルはジョブ単位。完了ジョブは結果への
  導線を持つ。
- **Why**: 利用者はすでにボード単位で「いまの評価」を把握している。タスクカードを横断
  マージすると、どの設定のジョブかが消える。
- **Change freedom**: ジョブ見出し（モデル名、開始時刻、折りたたみ）や完了後の畳み方は
  変更できる。レーン内カード UI は既存・presence と共存できる限り変更できる。
- **Why not**: 全ジョブのタスクを 1 枚のカンバンに混ぜると、設定境界が消える。
- **Revisit when**: ジョブ数が常時多く、縦積みが高さで破綻した時（タブ化等）。

### DEC-003: レート制限はプロセス全体で共有し、プロバイダ ID をキーにする

- **What**: すべての評価ジョブの LLM 呼び出し（被験・judge・holistic）は、発行直前に
  `provider_id` 単位の共有リミッタで許可を取る。ジョブ内セマフォや既存 judge pacing の
  上に、プロセス全体の上限が乗る。
- **Why**: ジョブごとに独立 pacing すると、同時 3 本で実効レートが 3 倍になり、制限の意味が
  消える。モデル ID ではなくプロバイダ ID で束ねるのは、キーと upstream クォータが
  プロバイダ単位だからである。
- **Change freedom**: 実装（sliding window / token bucket 等）、待ちの公平性、ログ粒度は
  窓内上限を守る限り変更できる。差し込み点は adapter 境界を原則とするが、同等の全局保証が
  あればエンジン側でもよい。
- **Why not**: フロントだけの throttle は API 直叩きや複数クライアントで迂回される。
- **Revisit when**: API キー単位やモデル単位のクォータがプロバイダより支配的になった時。

### DEC-004: 制限パラメータは「窓秒数あたりの最大リクエスト数」とし Settings で編集する

- **What**: 各プロバイダに `max_requests` と `window_seconds` を持つ。Settings から編集・
  保存できる。サーバが永続化し、実際の発行制御に使う。未設定や未知プロバイダは推奨
  デフォルト（保守的既定）を適用する。builtin プロバイダには推奨値を内蔵する。
- **Why**: 「秒間」と「数十秒窓」の両方を同じモデルで表せる。利用者の契約プラン差を
  Settings で吸収しつつ、初回は安全側デフォルトで動かす。
- **Change freedom**: 推奨デフォルトの具体数値、UI の入力単位表示、リセットボタン有無は
  変更できる。追加フィールド（burst 等）は後方互換を壊さない範囲で足せる。
- **Why not**: 固定定数のみだとプラン差に追随できない。RPS 浮動小数だけだと長窓の契約
  （例: 50 req / 60s）が表現しづらい。
- **Revisit when**: プロバイダが公式に RPM/TPM を機械可読で返し、自動同期したくなった時。

### DEC-005: レート待ちはキャンセル可能であり、待ち中もジョブ進捗として観測できる

- **What**: リミッタ待機中にジョブキャンセルが来たら待ちを中断する。可能なら「レート制限
  待ち」をジョブまたはタスク進捗に示せるようにする（初版は最小表示でもよいが、無言ハングに
  見せないことを目標とする）。
- **Why**: 同時実行時の最大の体感障害は「進まないが死んでもいない」状態だから。
- **Change freedom**: 表示粒度（ジョブバナー / タスクカード）は変更できる。
- **Why not**: 待ちを無視してキャンセル不能にすると、上限変更や誤設定時に操作不能になる。

## Consequences / Impact

- `runStore` は単一スロットからジョブ集合へ変わる。Layout の実行中インジケータも複数対応が
  必要になる。
- adapter / engine 経路に共有リミッタが入り、単体テストで時間制御（fake clock）が要る。
- Settings にレート制限セクションが増え、サーバ永続ファイルが増える。
- UI-Feat-61（presence）はジョブ内 active カードに載る前提で共存する。ボード全体気候には
  しない（既存 Intent と整合）。

## Quality Implications

- High risk: 同時実行 stampede、制限迂回、4 本目の拒否漏れ、1 ジョブ回帰。
- 制限遵守は決定的ユニットで窓内カウントを検証する。E2E 実 API は必須にしない。
- Manual QA で 2〜3 ジョブ縦積みと個別キャンセルを確認する。

## Intent-derived Invariants

- INV-001 (from DEC-001): 同一プロセスで同時に `running` な評価ジョブは最大 3 本である（開始時にサーバが保証）。
- INV-002 (from DEC-003): 評価ジョブの LLM 完了呼び出しは、`provider_id` キーのプロセス共有リミッタを経由する。
- INV-003 (from DEC-004): 設定未上書きのプロバイダには推奨デフォルト（または未知プロバイダ用の保守的既定）が適用され、制限なしの暗黙無限発行にならない。

## Rollback / Follow-ups

- rollback は active run registry、共有リミッタ、Settings 永続化、multi-job UI を同時に戻す。
- 保存済み結果 JSON の破壊は不要。単一ジョブ SSE クライアントは従来どおり動く状態へ戻す。
- follow-up 候補: 同時上限の Settings 化、永続キュー、タブ間調整、公式 RPM 自動同期。
