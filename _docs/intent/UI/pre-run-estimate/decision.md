# Intent: Pre-run cost and duration estimate

## Context

開始前に「この設定でいくら・何分かかりそうか」が見えない。履歴があれば信じたいが、
単一の最近傍だけだと外れ値と構成差に弱い。履歴が溜まるほど精密にしたい一方、被験モデルの
単価レジームまで別履歴で壊したくはない。

## Decisions

### DEC-001: 複数履歴を類似度で束ね、近いほど強く効かせる

- **What**: 実行前見積は、条件が近い履歴ほど大きく効くよう複数 run を合成する。遠い履歴も
  チャネルが許す限り捨てず、薄い寄与として残してよい。
- **Why**: 履歴が増えると近い観測に質量が集まり、見積が安定する。単一最近傍は外れ値 1 本に
  依存する。
- **Change freedom**: 類似度の定義、減衰の形、実装上の重み式は
  `_docs/reference/UI/pre-run-estimate/reference.md` 側で差し替えてよい。
- **Why not**: 等重み平均は遠い構成を過大に効かせる。完全一致のみは当たりが極端に減る。

### DEC-002: 所要の正典は「押してから終わるまで」の待ち時間

- **What**: 利用者向け所要の主値は wall-clock とする。処理時間合算は補助信号に留め、主表示と
  すり替えてはならない。
- **Why**: 開始前の判断対象は待ち時間であり、時間 ROI が扱う処理時間とは別質問だから。
- **Change freedom**: 補助信号の使い方・ラベルは変更できる。
- **Why not**: 処理時間合算を主値にすると並列待ち感とずれる。

### DEC-003: 構成が違う履歴は、混ぜる前に比べられる単位へ落とす

- **What**: タスク数・judge 数・run 回数が違う履歴を、構成無視のまま平均してはならない。
  一度「単位あたり」に揃えてから合成し、いまの計画構成へ戻す。
- **Why**: 負荷が違う run をそのまま平均すると、安い／短い構成の履歴が系統的に見積を歪める。
- **Change freedom**: 負荷の定義式、ラベル（履歴 / 履歴+構成補正）は変更できる。
- **Why not**: 総額・総所要を構成無視で平均すると、DEC-001 の「近さ」補正だけでは足りない。

### DEC-004: コスト不明を成功扱いにしない。所要だけ粗い推定を許す

- **What**: 使える履歴コストが無いとき、コストは不明（N/A）とする。所要だけは、履歴が無くても
  粗い構成ベースの推定を出してよい。
- **Why**: 誤った $0 はクレジット判断を誤らせる。待ち時間のオーダー感には、粗い値でも価値が
  ある。
- **Change freedom**: 粗い推定の具体式、将来の単価参照は変更できる。
- **Why not**: 単価不明を 0 円表示すると「無料そう」に読まれる。

### DEC-005: 実行前専用の表示にし、確度を隠さない

- **What**: 見積は idle（実行前）に出し、履歴由来か粗い推定か不明かをラベルで示す。
- **Why**: 実行中 ETA・事後コストと役割が違う。数値だけだと確定値に読まれる。
- **Change freedom**: 置き場所・文言・補助的な確度表示は変更できる。

### DEC-006: 被験コストは横断禁止。所要（と judge 側）は薄い横断を許す

- **What**: コストのうち被験に紐づく分は、別被験の履歴を混ぜてはならない。judge 側コストや
  wall-clock 所要は、別被験でも薄い寄与として使ってよい。
- **Why**: 被験コストは単価レジームそのものなので、横断すると安い／高い被験の履歴が系統的に
  引きずる。所要や judge 側は「同じ pipeline を回した気配」として共有できる。
- **Change freedom**: チャネル分割の実装、薄い寄与の強さ、judge 分の近似方法は変更できる。
  「被験コストを別被験で汚さない」「wall を完全に被験一致限定にしない」は維持する。
- **Why not**: 総額を丸ごと被験横断で混ぜると、DEC-004 の「不明を誤魔化さない」以前に、
  金額の意味が壊れる。wall を被験一致のみにすると、別被験で積んだ judge 待ちの気配を捨てる。

## Consequences / Impact

- 初回被検や価格不明モデルではコスト N/A が増える（意図的）。
- 別被験の薄い wall 寄与は、同一被験の近い履歴が少ない間だけ相対的に残る。
- 旧「単一最近傍」は DEC-001 の理由で置き換える。式の置き場は reference。

## Quality Implications

- 「複数履歴合成」「構成差の単位化」「コスト不明の非 0 埋め」「被験コスト非横断」を検証する。
- 重み式そのものの数値回帰は reference／実装テストの関心であり、Intent の契約ではない。

## Intent-derived Invariants

- INV-001 (from DEC-004): コスト見積が不明なとき、数値 0 を推定成功として表示してはならない。
- INV-002 (from DEC-006): 被験に紐づくコスト推定に、別被験の履歴を含めてはならない。

## Rollback / Follow-ups

- 複数履歴合成をやめ単一最近傍へ戻す場合は、DEC-001 の Why を覆す理由が先に必要。
- follow-up: summary の豊富化、単価カタログ、定数較正は reference／別タスク。
