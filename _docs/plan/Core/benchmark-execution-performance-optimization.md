## Overview
- 実行時間の主要因となっているタスク直列実行を解消し、評価ワークフロー全体の完了時間を短縮する。
- 速度改善を優先しつつ、既存の評価品質要件（特に Gemini の出力長要件）を維持する。

## Scope
- `app.py` のベンチマーク実行を、タスク単位で並列実行できるように変更する（初期並列度: 3）。
- `core/benchmark_engine.py` の judge 試行間にある固定待機を廃止し、必要時のみバックオフする方式に変更する。
- 各 judge モデル内の `judge_runs` を queue + worker 方式で並列実行する（初期 worker 数: 3）。
- 同一 judge モデルへの呼び出しは最小間隔 + ジッターで投入時刻をずらし、秒間バーストを抑制する。
- judge 実行で失敗累積が閾値に到達した場合、当該モデルの残り試行を早期スキップする（初期閾値: 2）。
- 進捗表示・キャンセル動作が並列実行後も破綻しないように整合性を確保する。

## Non-Goals
- `max_tokens` の削減やトークン上限最適化（Gemini の出力事情を踏まえて今回は対象外）。
- モデル一覧取得 (`core/model_catalog.py`) のキャッシュ TTL 導入や provider 取得並列化。
- UI の進捗更新頻度最適化（UX 影響があるため別フェーズで検討）。

## Requirements
- **Functional**
  - 複数タスク選択時、最大 3 タスクまで同時に評価できる。
  - 並列実行時も、各タスク結果の構造（`tasks[].judge_results`）を既存互換で保持する。
  - 各 judge モデル内で `judge_runs` を並列実行しつつ、同一モデルへの投入は最小 250ms 間隔 + 0-150ms ジッターで平準化する。
  - judge の失敗累積が 2 回に達した場合、同一 judge モデルの残試行を `skipped` として記録し、次の judge または次タスクへ進む。
  - 429 や一時的 API エラー時は既存のリトライ挙動を維持し、固定 sleep は入れない。
  - キャンセル時は進行中タスクを安全に中断し、完了済み分のみ結果に反映する。
- **Non-Functional**
  - 小規模ケース（タスク 1 件）で現行と同等の挙動を維持する。
  - 既存の結果表示 UI と保存フォーマットに追加の後方互換性問題を生じさせない。
  - 例外発生時もアプリ全体が停止せず、失敗情報を結果 JSON に残して継続可能とする。

## Tasks
- `run_benchmark` をタスク並列実行に対応させる（セマフォ制御: 3、順序付き収集、キャンセル整合）。
- `BenchmarkEngine._run_judge_evaluation` の固定 `sleep(0.5)` を削除し、リトライ時バックオフのみ残す。
- judge run を queue + worker で並列化し、投入ペーシング（最小間隔 + ジッター）を追加する。
- judge 失敗累積カウンタを導入し、閾値到達時の早期スキップ記録ロジックを追加する。
- 進捗カウント方式を、並列実行でも過不足なく進むように見直す。
- 必要なユニットテストを追加し、直列時との互換挙動を確認する。

## Test Plan
- タスク 1 件実行で、既存結果との差分がないことを確認する。
- タスク 4 件以上で、同時実行数が 3 を超えないことをログまたはモックで確認する。
- 同一 judge モデル内で run が並列実行されること、および投入時刻が分散されることを確認する。
- judge 失敗累積をモックで再現し、閾値到達後の残試行が `skipped` として記録されることを確認する。
- 固定 sleep 廃止後も、リトライ時のみバックオフ待機が発生することを確認する。
- キャンセル操作時に、完了済みタスクのみが結果に残り、未完了タスクでハングしないことを確認する。

## Deployment / Rollout
- 初期値は「タスク並列度 3 / judge-run worker 3 / 失敗閾値 2 / 250ms + jitter」で導入する。
- 実運用ログで API エラー率や失敗率の悪化がないか確認し、必要に応じて並列度を調整する。
- 問題発生時は、並列実行を無効化して直列モードに戻せるよう実装時に切替点を明確化する。
